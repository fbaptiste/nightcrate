# v0.40.3 — Plan a Night (pick any date): implementation design

Implementation notes for the v0.40.3 PLAN.md entry. Lets the Target Planner plan
for **any night**, not just tonight. Banked so a future session can build it
without re-exploring.

## Key finding — the backend is already date-parametric

A `date` query param already exists end-to-end; "tonight-only" is purely a
frontend omission (the store never sets `date`, so the backend defaults to
`tonight_date()`). **No backend compute work, no migration.** This is almost
entirely frontend plumbing to expose the date the backend already accepts.

Verified already-threaded:
- `api/planner.py` — `date_: date | None = Query(None, alias="date")` on
  `list_targets` (~L479), `score_single_target` (~L1046), `target_sky_track`
  (~L1140). Each resolves `night_date = date_ if date_ is not None else
  _tonight_date(tz)` and feeds it straight into compute.
- `services/planner_visibility.py` — `compute_visibility_snapshot(location,
  horizon, night_date, dsos)`; `_astro_dark_window(loc, night_date, …)`;
  `VisibilityCache._key()` already includes `night_date.isoformat()`
  (so distinct dates = distinct cache entries). `max_entries = 4` (LRU).
- `services/planner_scoring.py` — `score_targets` is snapshot-driven and
  date-agnostic; works for any date with zero change.
- `services/planner_sky_track.py` — `compute_sky_track(location, horizon,
  night_date, dso)`.
- `services/astronomy.py` — all consumed functions take explicit date/time;
  `tonight_date()` is the default-source only.
- `frontend/src/api/planner.ts` — `date` already typed + serialized on
  `fetchPlannerTargets` (param L153 / serialize L195), `fetchSingleTargetScore`
  (L266 / L281), `fetchSkyTrack(dsoId, locationId, horizonId?, date?)`
  (4th positional arg L291 / serialize L295).

## Locked decisions
1. **Tab rename** "Tonight from {Location}" → **"Plan a Night"**.
2. **Now-status hidden when date ≠ today** — drop the up/rising/set indicator
   AND its sort option (reuse the tonight-only-field hiding mechanism).
3. **Default date = always tonight** (today in the location's tz). **Ephemeral —
   NOT persisted**; the one planner control that does not go through
   `usePlannerSettingsSync`. Resets to tonight each session.
4. Ships as **v0.40.3** (interlude before v0.41.0).

## Frontend work

### 1. Store — `stores/plannerStore.ts`
Add `selectedDate: string | null` (ISO `YYYY-MM-DD`; `null` = tonight) and
`setSelectedDate`. **Do NOT** add it to `usePlannerSettingsSync.ts` (decision 3 —
ephemeral). Default `null`.

### 2. Date helper
Lift `todayInTimezone(tz)` out of `components/calculators/TonightCalc.tsx`
(~L55-74, currently local + unexported) into a shared `lib/` util, or duplicate
the ~18 lines. Returns today's `YYYY-MM-DD` in the location's IANA tz (handles
the evening rollover). Use it to seed the date input when `selectedDate` is null.

### 3. Controls bar — `pages/PlannerPage.tsx`
- Row-1 selector `<Stack>` is at ~L521. Insert a Date `<TextField type="date"
  size="small" label="Date" slotProps={{ inputLabel: { shrink: true } }}>`
  **before** the Location `FormControl` (~L522), gated `restrictTonight &&`.
  Value = `selectedDate ?? todayInTimezone(locationTz)`. `onChange` →
  `setSelectedDate` + reset pagination (`setPagination(p => ({...p, page: 0}))`),
  matching the sibling selectors.
- `targetsQuery` (~L234-299): add `date: restrictTonight ? selectedDate : null`
  to the `fetchPlannerTargets({…})` call AND add `selectedDate` to the query-key
  object (~L235-258).
- Pass `selectedDate` down to `<PlannerDetailPanel …>` (~L970-980).
- Tab label (~L378): change the dynamic `Tonight from {locationName}` to
  `"Plan a Night"`.

### 4. Now-status gating (decision 2)
- Sort option: `now_status` is `tonightOnly` in `lib/plannerSortFields.ts`. Extend
  `sortFieldAvailable` (or filter at the `PlannerSortPanel` / `PlannerPage` call
  site) to ALSO drop `now_status` when a non-today date is selected. Thread an
  `isToday`/date arg through.
- List indicator: wherever the now-status glyph renders (`PlannerTargetCard`),
  render it only when the selected date is today.

### 5. Detail panel — `components/planner/PlannerDetailPanel.tsx` (preview-override pattern)
Mirror the existing `previewLocationId/Horizon/Rig` pattern exactly:
- Add `previewDate` state seeded from the new `selectedDate` prop (~L157-163) and
  reset it in the `dsoId`-change effect (~L164-170).
- Render a date input in the `DialogTitle` right-hand `<Stack>` **above** the
  Location indicator (insert before ~L391), styled like the other borderless
  `variant="standard"` override controls.
- `previewScoreQuery` (~L227-245): add `date: previewDate` to the
  `fetchSingleTargetScore` params AND add `previewDate` to its query key.
- `skyTrackQuery` (~L209-219): pass `previewDate` as the 4th arg to
  `fetchSkyTrack(...)` AND add it to the key.
- `annualHoursQuery` (~L261-284): date-independent — leave alone.

### 6. API client — `frontend/src/api/planner.ts`
No change; the three fetchers already accept `date` (see Key finding). Just pass
the values + add to query keys.

## Optional
Bump `VisibilityCache.max_entries` (planner_visibility.py, currently 4) to ~8-12
if date-stepping thrashes the LRU — each new date is a full ~14k-DSO recompute
(sub-second but not free).

## Edge cases — already handled by per-date compute
Polar/seasonal no-astro-dark nights, southern hemisphere, far-future ephemeris,
and correct future moon phase/separation are all computed per `night_date`. Year
charts (Best Time of Year, annual hours) are date-independent and untouched.

## Verification

  updates; scores reflect that night's conditions (altitude, moon).
- Open a target → date carries over; change the per-item date → only that panel

- now-status indicator + sort option absent for a non-today date, present for today.
- Reload the app → date is back to tonight (ephemeral).
